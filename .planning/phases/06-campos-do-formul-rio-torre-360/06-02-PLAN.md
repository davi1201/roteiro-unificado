## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Usuário (browser) → IdentificacaoSection inputs | Valores entram via register/setValue do RHF; sincronizam para Zustand. Sem chamadas de rede nesta fase. |
| Zustand sectionData → sessionStorage | sectionData é serializado para `sessionStorage` com chave `form-data-${tenantId}` pelo subscriber existente. Multi-tab same-tenant compartilham; cross-tenant isolado. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-02-01 | Tampering | Campo CNPJ com máscara client-side | mitigate | Validação Zod com regex final `/^\\d{2}\\.\\d{3}\\.\\d{3}\\/\\d{4}-\\d{2}$/` aceita apenas o formato canônico. Persistência real (Phase 8) deve revalidar server-side via Supabase. Aceito que a máscara client-side é UX, não segurança. |
| T-06-02-02 | Information Disclosure | sectionData no sessionStorage | accept | sessionStorage é por-aba e tenant-namespaced (chave `form-data-${tenantId}`). Risco: outra aba do mesmo tenant lê dados parciais. Aceito porque é o mesmo usuário e o mesmo tenant; comportamento intencional de retomada de rascunho. |
| T-06-02-03 | Tampering | watch + useEffect loop | mitigate | Loop infinito documentado em T-05-04-04 (FormLayout.tsx linha 42). Mitigação: `store` nunca entra nas deps; eslint-disable-next-line `react-hooks/exhaustive-deps` documentado no comentário acima do useEffect. Verificado via acceptance_criteria. |
| T-06-02-04 | Validation bypass | Cast de `defaultValues as Partial<IdentificacaoData>` | accept | Cast necessário porque `sectionData[tab]` é `Record<string, unknown>` (D-02). Runtime é seguro: valores vieram de submissões anteriores no mesmo schema; Zod re-valida em qualquer onBlur subsequente. |
| T-06-02-SC | Tampering | Pacotes npm | accept | Nenhum pacote novo nesta fase. Stack RHF/Zod/Zustand já auditado. |

# Phase 08.1: Code Review Report

**Reviewed:** 2026-05-24
**Depth:** standard
**Files Reviewed:** 1
**Status:** issues_found

## Summary

This migration replaces the `assessments_update_draft` RLS policy to fix SAVE-03: the original `WITH CHECK (status = 'draft')` blocked the `draft→submitted` transition because PostgreSQL evaluates `WITH CHECK` against the new (post-UPDATE) row, not the old one.

The core logic of the fix is **correct**. Keeping `USING (status = 'draft')` restricts which rows a company user can target (only drafts), while expanding `WITH CHECK` to `status IN ('draft', 'submitted')` allows the new row to be either an autosave (draft) or a submission (submitted). The `submitted→draft` regression path is properly blocked by `USING`. The `DROP POLICY IF EXISTS` + `CREATE POLICY` pair is transactional in Supabase's migration runner, so there is no window of denied access. Idempotency is sound.

Two security/data-integrity concerns were found at warning level. Two informational gaps in surrounding migrations are called out as context (not introduced by this migration, but related enough to be worth noting here).

## Warnings

### WR-01: Policy name `assessments_update_draft` is now semantically incorrect

**File:** `supabase/migrations/20260523000002_fix_rls_assessments_submit.sql:20`

**Issue:** After this migration, the policy named `assessments_update_draft` permits `status = 'submitted'` in its `WITH CHECK` clause. The name implies the policy governs only draft rows, which is now only half-true (USING is draft-restricted; WITH CHECK is not). Any engineer reading the policy name in a `\dp assessments` output or in a future audit will conclude it blocks non-draft outcomes — which is wrong. This creates a maintainability hazard: the next migration touching this policy may re-introduce the bug under the false assumption that the name reflects the full constraint.

**Fix:** Rename the policy to something that reflects both halves of its intent, e.g., `assessments_update_company` or `assessments_company_update_draft_or_submit`. Since PostgreSQL does not support `RENAME POLICY`, a new DROP+CREATE pair would be needed:

```sql
DROP POLICY IF EXISTS "assessments_update_draft" ON public.assessments;

CREATE POLICY "assessments_company_update"
  ON public.assessments FOR UPDATE
  TO authenticated
  USING (
    (SELECT public.is_org_member(org_id))
    AND status = 'draft'
  )
  WITH CHECK (
    (SELECT public.is_org_member(org_id))
    AND status IN ('draft', 'submitted')
  );
```

At minimum, the existing comment block should be updated to clearly state that the name is historical and that `WITH CHECK` permits submission.

### WR-02: `submitted_at` timestamp is not constrained by RLS — company user can submit with arbitrary or NULL timestamp

**File:** `supabase/migrations/20260523000002_fix_rls_assessments_submit.sql:27-30`

**Issue:** The `WITH CHECK` clause only verifies `is_org_member(org_id)` and `status IN ('draft', 'submitted')`. It places no constraint on the `submitted_at` column. When a company user executes the submit UPDATE, they can set `submitted_at` to any value — including `NULL`, a past date (e.g., `'2020-01-01'`), or a future date — without RLS blocking it. The dashboard and audit trail depend on `submitted_at` being the real server-side timestamp of submission. A company user manipulating this field can falsify the submission timestamp record.

The blast radius is limited to the user's own organisation (they cannot affect other orgs' records), but the integrity of audit data within that org is compromised.

**Fix:** The cleanest solution is a database trigger that enforces `submitted_at = NOW()` whenever `status` transitions to `'submitted'`, removing the column from the client-controlled UPDATE entirely:

```sql
CREATE OR REPLACE FUNCTION public.set_submitted_at()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = ''
AS $$
BEGIN
  IF NEW.status = 'submitted' AND OLD.status = 'draft' THEN
    NEW.submitted_at := NOW();
  END IF;
  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_assessments_set_submitted_at
  BEFORE UPDATE ON public.assessments
  FOR EACH ROW
  EXECUTE FUNCTION public.set_submitted_at();
```

A lighter alternative is to add `submitted_at IS NOT NULL` and `submitted_at <= NOW()` constraints to the `WITH CHECK`, but a trigger provides stronger guarantees and removes the attack surface entirely.

## Info

### IN-01: INSERT policy does not constrain `status` — company user can INSERT directly with `status = 'submitted'`

**File:** `supabase/migrations/20260522000008_rls_policies_assessments.sql:29-34` (context — not the reviewed file)

**Issue:** The `assessments_insert` policy's `WITH CHECK` only verifies `is_org_member(org_id)`. It does not constrain `status`. A company user can therefore issue an INSERT with `status = 'submitted'` directly, bypassing the draft workflow, the autosave unique-index guard, and the `submitted_at` enforcement (if added). This gap predates the reviewed migration but is closely related to the SAVE-03 fix and worth addressing in the same remediation window.

**Fix:** Add `AND status = 'draft'` to the `assessments_insert` WITH CHECK:

```sql
-- In a follow-up migration or alongside WR-02 trigger work:
DROP POLICY IF EXISTS "assessments_insert" ON public.assessments;

CREATE POLICY "assessments_insert"
  ON public.assessments FOR INSERT
  TO authenticated
  WITH CHECK (
    (SELECT public.is_org_member(org_id))
    AND status = 'draft'
  );
```

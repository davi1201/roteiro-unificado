# Phase 08.1: Fix SAVE-03 RLS — permitir draft→submitted - Pesquisa

**Pesquisado:** 2026-05-23
**Domínio:** Supabase RLS (Row Level Security) — UPDATE policies com WITH CHECK
**Confiança:** HIGH

---

## Sumário

O requisito SAVE-03 ("usuário pode submeter avaliação — status muda para `submitted` imutável") está bloqueado por um bug de RLS em produção. A política `assessments_update_draft` em `supabase/migrations/20260522000008_rls_policies_assessments.sql` foi escrita com `WITH CHECK (is_org_member(org_id) AND status = 'draft')`. Segundo a semântica PostgreSQL confirmada na documentação oficial, `WITH CHECK` avalia a **nova linha** após o UPDATE. Quando `useSubmitAssessment` executa `UPDATE assessments SET status = 'submitted'`, a nova linha tem `status = 'submitted'`, violando o `WITH CHECK` — PostgreSQL retorna um erro RLS e a mutação cai no `onError`.

O fix é cirúrgico: uma nova migration substitui a política `assessments_update_draft` por uma versão corrigida cujo `WITH CHECK` permite `status IN ('draft', 'submitted')` para membros da org. A política mantém o `USING (status = 'draft')` para garantir que apenas drafts possam ser atualizados — ou seja, a transição `submitted→draft` continua proibida. Nenhum código TypeScript precisa mudar — o hook `useSubmitAssessment.ts` já implementa o comportamento correto; é o banco que bloqueia a operação.

Escopo desta fase: 1 arquivo SQL novo (migration) + atualização dos testes de integração para verificar o comportamento end-to-end real (sem mocks de Supabase).

**Recomendação principal:** Criar `supabase/migrations/20260523000002_fix_rls_assessments_submit.sql` que dropa e recria `assessments_update_draft` com `WITH CHECK (is_org_member(org_id) AND status IN ('draft', 'submitted'))`.

## Diagnóstico Técnico Detalhado

### Raiz do Problema

**Arquivo:** `supabase/migrations/20260522000008_rls_policies_assessments.sql`, linhas 38–48

```sql
-- ATUAL (bloqueado para submissão):
CREATE POLICY "assessments_update_draft"
  ON public.assessments FOR UPDATE
  TO authenticated
  USING (
    (SELECT public.is_org_member(org_id))
    AND status = 'draft'
  )
  WITH CHECK (
    (SELECT public.is_org_member(org_id))
    AND status = 'draft'   -- ← avalia a NOVA linha → falha quando status vira 'submitted'
  );
```

**Semântica PostgreSQL confirmada** [CITED: postgresql.org/docs/current/sql-createpolicy.html]:

> "Note that the _check_expression_ is evaluated against the proposed **new contents** of the row, not the original contents."

Portanto, `WITH CHECK (status = 'draft')` avalia a linha resultante do UPDATE. Quando `useSubmitAssessment` faz `UPDATE SET status = 'submitted'`, a nova linha tem `status = 'submitted'` → WITH CHECK falha → PostgreSQL retorna RLS violation → `onError` no hook.

### Comportamento Observável

| Ação | USING (filtra linha) | WITH CHECK (valida nova linha) | Resultado |
|------|---------------------|-------------------------------|-----------|
| Autosave (`status: 'draft'`) | `status = 'draft'` ✓ | `status = 'draft'` ✓ | Sucesso |
| Submissão (`status: 'submitted'`) | `status = 'draft'` ✓ | `status = 'submitted'` ✗ | **RLS violation** |
| Regressão (`submitted→draft`) | `status = 'draft'` ✗ | — | Bloqueado (não chega ao CHECK) |

### Por que `assessments_update_admin` não resolve

A política `assessments_update_admin` existe mas apenas cobre o role `admin` (verifica `is_admin()`). Usuários `company` não são admins — a política admin não se aplica a eles.

### Padrão 1: DROP + CREATE POLICY em Migration Corretiva

**O quê:** Dropar a política quebrada e recriar com WITH CHECK corrigido.

**Por que DROP e não CREATE OR REPLACE:** PostgreSQL não suporta `CREATE OR REPLACE POLICY` — o comando não existe. É necessário `DROP POLICY` + `CREATE POLICY`. [CITED: postgresql.org/docs/current/sql-createpolicy.html]

**Quando usar:** Toda vez que uma policy RLS precisa de correção de semântica.

**Exemplo:**
```sql
-- Source: postgresql.org/docs/current/sql-createpolicy.html (semântica WITH CHECK confirmada)
-- e supabase.com/docs/guides/database/postgres/row-level-security

-- Dropar a policy atual (necessário — PostgreSQL não tem CREATE OR REPLACE POLICY)
DROP POLICY IF EXISTS "assessments_update_draft" ON public.assessments;

-- Recriar com WITH CHECK corrigido:
-- USING: só permite selecionar e modificar linhas onde status='draft' (protege submitted de ser re-editado)
-- WITH CHECK: permite que a nova linha tenha status='draft' (autosave) OU status='submitted' (submissão)
-- Isso bloqueia submitted→draft mas permite draft→submitted.
CREATE POLICY "assessments_update_draft"
  ON public.assessments FOR UPDATE
  TO authenticated
  USING (
    (SELECT public.is_org_member(org_id))
    AND status = 'draft'
  )
  WITH CHECK (
    (SELECT public.is_org_member(org_id))
    AND status IN ('draft', 'submitted')
  );
```

### Anti-Padrões a Evitar

- **Mover submissão para Edge Function com service_role:** Tecnicamente funciona (service_role bypassa RLS), mas é arquiteturalmente errado — adiciona complexidade (Deno, deploy separado, JWT) para resolver o que é uma configuração incorreta de policy. O fix correto é na policy.
- **Remover `WITH CHECK` completamente:** Perigoso — sem `WITH CHECK`, o USING expression é reutilizado para verificação da nova linha. Com `USING (status = 'draft')`, isso ainda bloquearia a submissão.
- **Usar `WITH CHECK (is_org_member(org_id))` sem restrição de status:** Permitiria que usuários company criassem linhas com qualquer status via UPDATE, incluindo transições arbitrárias. Mais permissivo que o necessário.
- **Alterar migration existente (20260522000008):** Migrations já aplicadas ao banco remoto são imutáveis — alterar o arquivo não re-executa no banco. Sempre criar nova migration.

## Armadilhas Comuns

### Armadilha 1: Confundir USING com WITH CHECK em UPDATE

**O que dá errado:** Desenvolvedores assumem que `USING (status = 'draft')` já define todo o comportamento do UPDATE — incluindo o que a nova linha pode conter.

**Por que acontece:** Em SELECT, só existe USING. Em INSERT, só existe WITH CHECK. UPDATE é o único que usa os dois com semânticas diferentes.

**Como evitar:** Para UPDATE:
- `USING` = qual linha existente pode ser acessada (OLD row)
- `WITH CHECK` = o que a nova linha pode conter (NEW row)
- Se `WITH CHECK` é omitido, PostgreSQL reutiliza o `USING` expression para ambos — o que neste caso ainda bloquearia a submissão.

**Sinais de alerta:** Erro PostgreSQL `42501: new row violates row-level security policy for table "assessments"` no console do Supabase ou no DevTools Network.

### Armadilha 2: Testar com mocks e achar que RLS está OK

**O que dá errado:** Testes unitários de `useSubmitAssessment` mockam o cliente Supabase — nunca executam de fato contra o banco. Os 4 testes `SAVE-03` passam mesmo com a policy quebrada.

**Por que acontece:** É a natureza dos mocks — exercitam apenas a lógica TypeScript, não o comportamento do banco.

**Como evitar:** Após aplicar a migration, verificar manualmente no Supabase Dashboard > Table Editor ou via `supabase db reset` + seed + teste end-to-end no browser.

### Armadilha 3: `CREATE OR REPLACE POLICY` não existe no PostgreSQL

**O que dá errado:** Tentar `CREATE OR REPLACE POLICY` gera syntax error — o comando não existe no PostgreSQL (ao contrário de funções e views). [CITED: postgresql.org/docs/current/sql-createpolicy.html]

**Como evitar:** Sempre usar `DROP POLICY IF EXISTS "nome" ON tabela;` seguido de `CREATE POLICY "nome" ...`.

### Armadilha 4: Alterar migration existente em vez de criar nova

**O que dá errado:** Editar `20260522000008_rls_policies_assessments.sql` não tem efeito no banco remoto — migrations já aplicadas não são re-executadas automaticamente.

**Como evitar:** Sempre criar nova migration com timestamp posterior. O banco remoto aplica apenas migrations novas.

## Exemplos de Código

### Migration Corretiva Completa

```sql
-- Source: postgresql.org/docs/current/sql-createpolicy.html (WITH CHECK avalia NEW row)
-- File: supabase/migrations/20260523000002_fix_rls_assessments_submit.sql

-- Fix SAVE-03: permitir transição draft→submitted para role company.
--
-- Problema: a policy original tinha WITH CHECK (status = 'draft').
-- PostgreSQL avalia WITH CHECK contra a NOVA linha (após UPDATE).
-- Quando useSubmitAssessment seta status='submitted', a nova linha falha no CHECK.
--
-- Fix: manter USING restrito a 'draft' (só drafts podem ser atualizados)
-- mas ampliar WITH CHECK para aceitar 'submitted' como destino válido.
-- Transição submitted→draft continua proibida (USING bloqueia — linha já não é draft).

DROP POLICY IF EXISTS "assessments_update_draft" ON public.assessments;

CREATE POLICY "assessments_update_draft"
  ON public.assessments FOR UPDATE
  TO authenticated
  USING (
    (SELECT public.is_org_member(org_id))
    AND status = 'draft'
  )
  WITH CHECK (
    (SELECT public.is_org_member(org_id))
    AND status IN ('draft', 'submitted')
  );
```

### Padrões de Ameaça Conhecidos para RLS Supabase

| Padrão | STRIDE | Mitigação Padrão |
|--------|--------|-----------------|
| Escalada de privilégio via UPDATE (submeter avaliação de outra org) | Spoofing/Tampering | `USING (is_org_member(org_id))` garante que apenas rows da própria org são acessíveis |
| Regressão de status (submitted→draft) | Tampering | `USING (status = 'draft')` bloqueia — linha submitted não é visível para UPDATE |
| Company user acessando dados de outra org | Information Disclosure | Coberto por `assessments_select` e `assessments_update_draft` via `is_org_member` |

**Verificação de segurança do fix:** O fix não abre brecha de segurança porque:
1. `USING (status = 'draft')` — company só consegue fazer UPDATE em drafts da própria org (regressão `submitted→draft` permanece bloqueada)
2. `WITH CHECK (is_org_member(org_id) AND status IN ('draft', 'submitted'))` — company não pode definir status para um valor fora do enum, nem acessar org_id de outra organização

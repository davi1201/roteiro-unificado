<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → PostgREST | anon key atravessa aqui; RLS é o único controle de acesso |
| migration files → Postgres | SQL de migration executa com privilégio de superuser; não há validação adicional |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | org_members.user_id lookup in RLS | mitigate | Usar `(SELECT auth.uid())` com parênteses — não `auth.uid()` bare — em todos os USING/WITH CHECK; caching por statement reduz superfície de timing attack |
| T-02-02 | Elevation of Privilege | org_members.role write | mitigate | RLS policies de INSERT/UPDATE em org_members restritas a `is_admin()` — company users não podem elevar próprio role |
| T-02-03 | Spoofing | anon key access | mitigate | Todas as policies usam `TO authenticated` — anon role retorna 0 linhas em todas as tabelas |
| T-02-04 | Tampering | SECURITY DEFINER function search_path | mitigate | Todas as definer functions incluem `SET search_path = ''` — previne search_path hijacking |
| T-02-05 | Information Disclosure | is_org_member(org_id) RPC enumeration | accept | Função aceita UUID externo mas só verifica o user_id atual — não expõe membros de outras orgs; risco de membership existence check é aceito dado escopo de piloto |
| T-02-SC | Tampering | Supabase CLI install via brew | mitigate | CLI instalado de supabase/tap/supabase (tap oficial); verificar `supabase --version` após install antes de executar migrations |
</threat_model>

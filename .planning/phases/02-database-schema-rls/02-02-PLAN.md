<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| company role → assessments UPDATE | Policy deve bloquear UPDATE de registros submitted — somente draft é editável |
| admin role → org_members | Somente admin pode adicionar/remover membros; company não pode se auto-elevar |
| seed.sql → auth.users | Senhas em plaintext no seed file; apenas para ambiente de desenvolvimento/teste |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-P01 | Information Disclosure | RLS SELECT policy sem USING | mitigate | Toda tabela tem SELECT policy explícita com USING; sem SELECT policy = UPDATE silenciosamente bloqueado (Pitfall 3) |
| T-02-P02 | Tampering | assessments UPDATE por company em submitted rows | mitigate | assessments_update_draft policy inclui `AND status = 'draft'` em USING e WITH CHECK — submitted rows são imutáveis para company role |
| T-02-P03 | Tampering | org_members role elevation (self-promotion) | mitigate | INSERT/UPDATE em org_members restritos a is_admin() — company role não pode criar ou modificar memberships |
| T-02-P04 | Spoofing | Senhas de seed em repositório | accept | Senhas são de desenvolvimento/teste apenas (Admin@123, Empresa@123); documentar claramente no seed.sql; não usar em produção |
| T-02-P05 | Elevation of Privilege | is_admin() function exposure via RPC | accept | Função não aceita parâmetro user_id — só verifica current user; membership enumeration não é possível via este RPC |
  </threat_model>

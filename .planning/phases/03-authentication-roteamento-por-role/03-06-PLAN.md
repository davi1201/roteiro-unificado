  <behavior>
    Após supabase.auth.signInWithPassword bem-sucedido:
    - O onAuthStateChange do AuthProvider dispara e popula role e orgId no context
    - Login.tsx deve aguardar o role estar disponível antes de redirecionar
    - Estratégia: após signInWithPassword sem erro, ler role e orgId do context (já atualizados pelo onAuthStateChange)
    - Se role === 'admin': navigate('/admin/dashboard')
    - Se role === 'company' e orgId existe: navigate(`/form/${orgId}`)
    - Se orgId é null após login bem-sucedido (org_members não tem registro): exibir toast.error genérico — edge case de configuração incompleta

    Problema de timing: onAuthStateChange é assíncrono. Solução: no onSubmit, após signInWithPassword sem erro, aguardar uma iteração do event loop com um useEffect que observa mudanças em role/orgId, OU simplesmente usar navigate dentro do onSubmit após verificar role/orgId já presentes via re-read do context.

    Solução recomendada: no onSubmit, após signInWithPassword sem erro, checar diretamente — o AuthProvider atualiza estado sincronicamente no onAuthStateChange antes de qualquer re-render; role e orgId já estarão atualizados quando o onSubmit continuar. Se ainda null, usar setTimeout(() => { ... redirect ... }, 0) como fallback de única iteração.

    Logout (D-10): signOut já existe no AuthProvider e limpa sessão. O ProtectedRoute detecta a remoção da sessão via onAuthStateChange e redireciona para /login sem toast — este comportamento já está implementado nos planos anteriores. Login.tsx não precisa tratar logout.

    Usuário já autenticado acessando /login: usar useEffect ao montar para verificar se session já existe (via useAuth) e redirecionar diretamente, sem exibir o form.
  </behavior>

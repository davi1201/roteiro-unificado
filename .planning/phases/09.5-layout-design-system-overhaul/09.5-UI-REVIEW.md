## 🚨 Blockers

### BLOCKER 1 — Footer renderiza como 3ª coluna flex no desktop

**Arquivo:** `roteiro-unificado/src/features/form/FormLayout.tsx:217`

A estrutura atual:
```jsx
<div className="flex flex-1 flex-col md:flex-row">
  <aside>...</aside>      {/* coluna 1 */}
  <main>...</main>        {/* coluna 2 */}
  <div sticky bottom-0>  {/* coluna 3 — BUG */}
```

Em `md:flex-row`, o footer vira uma terceira coluna estreita à direita do formulário — não uma barra na base do `<main>`. O `sticky bottom-0` não tem efeito neste contexto. Os botões "Próxima aba →" e "Enviar Avaliação" ficam funcionalmente mal posicionados.

**Fix:** Mover o footer `<div>` para dentro do `<main>` antes do `</main>`, e trocar `pb-24` por padding menor.

---

### BLOCKER 2 — Form sidebar não é sticky; some em sections longas

**Arquivo:** `roteiro-unificado/src/features/form/FormLayout.tsx:180`

A `<aside>` é um flex filho estático. Em `TorreSiengeSection` (12 módulos), `TorreAcessoSection`, e `HabRepositoriosSection` (14 domínios), o conteúdo tem 800–1200px. Após scrollar ~300px, a navegação de abas desaparece completamente.

O `AdminSidebar` usa `position: fixed` e permanece visível. A form sidebar deve ter pelo menos `sticky`.

**Fix:**
```jsx
<aside className="md:sticky md:top-0 md:h-screen md:overflow-y-auto ...">
```

## ⚠️ Warnings

### WARNING 1 — Padding do conteúdo: Form p-4/p-6 vs Admin p-8

**Arquivo:** `roteiro-unificado/src/features/form/FormLayout.tsx:198`

Admin `<main>` usa `p-8` (32px). Form `<main>` usa `p-4 pb-24 md:p-6 md:pb-24` (16–24px). Diferença de 8–16px em todos os lados — causa concreta do "formulário não segue o mesmo padrão visual do admin".

**Fix:** `p-6 pb-28 md:p-8 md:pb-32` para alinhar com o admin.

---

### WARNING 2 — FormCard inside cols=2 gera altura desigual sem items-start

**Arquivo:** `roteiro-unificado/src/features/form/FormCardRow.tsx:20-30`

Quando uma FormCardRow tem cols=2 e os cards têm alturas diferentes (ex: card com textarea vs card com 3 inputs), o card menor estica para preencher a linha. Sem `items-start` no grid, isso cria espaço branco vazio no body do card menor.

**Fix:** Adicionar `items-start` ao grid do FormCardRow.

---

### WARNING 3 — "Salvar rascunho" é ghost CTA

**Arquivo:** `roteiro-unificado/src/features/form/FormLayout.tsx:240`

O botão "Salvar rascunho" executa apenas `toast.info('Rascunho será salvo automaticamente em 1.5s')` — não dispara save. Label implica ação mas handler não faz nada.

**Fix:** Mudar para label estático "Autosalvo ativo" ou disparar save real.

---

### WARNING 4 — Sidebar do formulário usa largura arbitrária, não o token

**Arquivo:** `roteiro-unificado/src/features/form/FormLayout.tsx:180`

Sidebar usa `md:max-w-[300px] md:min-w-[220px]` em vez do token `--sidebar-width: 240px`. AdminSidebar usa `w-60` (240px). As sidebars têm larguras diferentes em desktop.

**Fix:** Usar `md:w-60` (que usa o mesmo valor do `--sidebar-width` token).

---

### WARNING 5 — AutosaveIndicator mostra "Não salvo" ao hidratar rascunho existente

**Arquivo:** `roteiro-unificado/src/stores/formStore.ts` + `useAutosave.ts`

`lastSavedAt` é resetado para `null` no `onRehydrateStorage`. Ao abrir um rascunho salvo, o indicador mostra "Não salvo" até o primeiro keystroke disparar autosave. UX enganosa — o usuário não sabe se o rascunho carregou.

**Fix:** Ao hidratar `formStore` com dados existentes, definir `lastSavedAt` como data de hoje (ou ler timestamp do localStorage).

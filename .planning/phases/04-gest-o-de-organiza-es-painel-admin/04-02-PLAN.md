    Tipo exportado:

    ```
    }
    ```

    Hook:
    ```
    export function useOrgs() {
      return useQuery<OrgWithMemberCount[]>({
        queryKey: ['orgs'],
        queryFn: async () => {
          const { data, error } = await supabase
            .from('orgs')
            .select('id, name, cnpj, active, created_at, org_members(count)')
            .order('created_at', { ascending: false })
          if (error) throw error
          // Normaliza o shape: PostgREST retorna org_members como array com objeto { count }
          return (data ?? []).map((row) => ({
            id: row.id,
            name: row.name,
            cnpj: row.cnpj,
            active: row.active,
            created_at: row.created_at,
            member_count: Array.isArray(row.org_members) && row.org_members.length > 0
              ? (row.org_members[0] as { count: number }).count
              : 0,
          }))
        },
      })
    }
    ```

    Não usar paginação server-side; piloto é 5 orgs (per CONTEXT.md). Paginação client-side é tarefa da página.

    `queryKey: ['orgs']` é a key canônica que Plans 03 e 05 invalidarão.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-04-06 | Information Disclosure | useOrgs fetcha todas as orgs incluindo arquivadas | accept | Admin tem permissão para ver todas as orgs (ORG-03, RLS is_admin() → true para role admin). Não vaza dados — admin já é privilegiado. |

| T-04-09 | Denial of Service | Listagem cresce indefinidamente sem paginação server-side | accept | Piloto = 5 orgs (per CONTEXT.md §boundary). Paginação client-side é suficiente. Se escalar > 100 orgs, refatorar para server-side em fase de polimento. |
| T-04-10 | Spoofing | Click em linha navega para orgId arbitrário | mitigate | orgId vem direto do payload Supabase (validado por RLS). Não há injeção de URL via input do usuário. Página de detalhe (Plan 04) revalida acesso via query da própria org. |

    2. Mutation:
       ```
       const queryClient = useQueryClient()
       const toast = useToast()

       const createOrgMutation = useMutation({
         mutationFn: async (data: CreateOrgFormData) => {
           const { error } = await supabase.from('orgs').insert({
             name: data.name,
             cnpj: data.cnpj,
             active: true,
           })
           if (error) throw error
         },
         onSuccess: () => {
           queryClient.invalidateQueries({ queryKey: ['orgs'] })
           toast.success('Organização criada com sucesso')
           reset()
           onClose()
         },
         onError: () => {
           toast.error('Erro ao criar organização. Tente novamente.')
         },
       })
       ```

    3. Submit handler:
       `const onSubmit = (data: CreateOrgFormData) => createOrgMutation.mutate(data)`

       <Dialog open={open} onClose={onClose}>

           <DialogTitle>Nova organização</DialogTitle>
           <DialogDescription>

           </DialogDescription>
         </DialogHeader>

         <form onSubmit={handleSubmit(onSubmit)} noValidate>
           <DialogContent className="flex flex-col gap-4">
             <div className="flex flex-col gap-1">
               <label htmlFor="org-name" className="text-sm text-gray-700">Nome da organização</label>
               <Input
                 id="org-name"

               <Input

               />

             </Button>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form → Supabase INSERT | Input do admin (name, cnpj) cruza para o backend via PostgREST |
| Zod validation → Supabase | Validação client-side é defesa primária; RLS + check constraints são defesa secundária |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-11 | Tampering | Admin bypassa validação Zod via DevTools e envia CNPJ inválido | mitigate | Validação Zod onBlur cobre 99% dos casos. Defense-in-depth: schema do Postgres permite `cnpj` como text (sem CHECK constraint nesta fase); validação adicional pode ser adicionada em policy/trigger em fase futura. Risco aceito para piloto (apenas admins internos têm acesso). |
| T-04-12 | Tampering | Admin tenta INSERT direto via console com active=false ou outras flags | accept | Admin tem permissão ampla via RLS is_admin() (per Phase 2). Não é vetor de ataque relevante — admin já é privilegiado e auditável via logs do Supabase. |
| T-04-13 | Information Disclosure | toast.error revela mensagem de erro do Supabase | mitigate | onError handler usa mensagem genérica `"Erro ao criar organização. Tente novamente."` — não expõe `error.message` cru. Detalhes ficam em `console.error` (Plan 02 já estabeleceu padrão). |
| T-04-14 | Repudiation | Admin cria org e nega ter feito | accept | Supabase logs registram `auth.uid()` da request via RLS context. Não há requirement de audit log explícito na Phase 4. |
| T-04-15 | Spoofing | Cliente envia name vazio ou whitespace-only que passa pela validação | mitigate | Schema usa `.trim().min(1, ...)` no name — strings whitespace-only falham na validação min(1) após trim. |
| T-04-16 | Denial of Service | Admin envia múltiplos submits rapidamente criando duplicatas | mitigate | Botão submit usa `isLoading={createOrgMutation.isPending}` → desabilitado durante submit. Duplicação requer cliques no botão Descartar entre submits — comportamento aceitável. |
</threat_model>
